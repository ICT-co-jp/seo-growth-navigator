# WordPressサイト改ざん事例まとめ｜被害パターン5種と発覚後の対処手順

「自分のサイトが攻撃される可能性は低い」と考えていても、WordPressサイトへの攻撃は中小企業や個人事業主のサイトにも日常的に発生しています。本記事では、実際に報告・報道された改ざん被害の事例を5つのパターンに整理し、被害の発覚サインから発覚後の対処手順、そして再発防止策までを体系的に解説します。自社サイトのリスクを正確に把握し、今日から取り組める対策のヒントとしてご活用ください。

---

## H2-1: WordPressサイトが攻撃を受けやすい理由

WordPressが攻撃者に狙われる背景を知ることは、対策を講じる上での第一歩です。「なぜWordPressなのか」を理解することで、守るべきポイントが自然と見えてきます。

### H3-1-1: 世界で広く使われるCMSだからこそ狙われる

WordPressが攻撃者に狙われる根本的な理由は、その圧倒的な普及規模にあります。WordPressは世界的に最も広く使われているCMSであり、個人ブログから大手企業サイト、官公庁・教育機関まで、あらゆる規模・用途のサイトで採用されています。

攻撃者の視点に立てば、WordPressを標的にすることは極めて「効率的」な選択です。1種類の攻撃手法を開発するだけで、世界中の膨大な数のサイトに同一の攻撃を仕掛けられるからです。さらに、WordPress向けのプラグインは公式ディレクトリ（wordpress.org/plugins/）だけで6万3千本以上が公開されており、これらのプラグインの脆弱性を突く攻撃も日常的に発生しています。

つまり「WordPressを使っているだけで攻撃対象になりやすい」という現実は、サイト管理者が常に意識しておくべき前提条件です。

### H3-1-2: 攻撃者が悪用する主な侵入経路の全体像

WordPressへの不正侵入は、大きく以下の4つの経路から発生します。

| 侵入経路 | 概要 |
|---|---|
| **プラグイン・テーマの脆弱性** | 未更新または品質の低いプラグイン・テーマに存在するセキュリティ上の欠陥（XSS、SQLインジェクション、ファイルアップロード不備等）を悪用する |
| **管理画面へのブルートフォース攻撃** | ログインページに対して、大量のIDとパスワードの組み合わせを自動で試し続けて管理者権限を奪取しようとする |
| **フィッシング・クレデンシャルスタッフィング** | 他のサービスから流出した認証情報（IDとパスワードの組み合わせ）を使い回すことでログインに成功する |
| **サーバー・ホスティング環境の脆弱性** | WordPressが動作するサーバーソフトウェア（PHPのバージョン、Webサーバー設定等）の不備を突いて侵入する |

これらの経路はそれぞれ独立しているわけではなく、「プラグインの脆弱性を悪用して管理者権限を奪取し、さらにサーバー内に不正ファイルを設置する」といった複合的な攻撃として現れることも多くあります。次章では、こうした攻撃が成功した場合にどのようなサインが現れるかを確認します。

## H2-2: 改ざん被害のサイン｜こんな症状が出たら要注意

改ざんが発生してから発覚するまでの時間が長ければ長いほど、被害は拡大します。典型的な異常サインを事前に把握しておくことで、早期発見・早期対処につなげましょう。

### H3-2-1: 自分でチェックできる7つの異常サイン

サイト管理者が日常的な確認の中で気づきやすい異常サインを以下に示します。

1. **知らないページや投稿が増えている** — 管理画面の「投稿一覧」「固定ページ一覧」に、自分が作成した覚えのないページが存在する。スパムコンテンツや誘導ページが作成されている場合があります。
2. **サイトのコンテンツが書き換えられている** — トップページや記事本文に、見覚えのない文字列・リンク・広告が挿入されている。または既存のコンテンツが全く別のページにすり替えられている。
3. **見知らぬリダイレクトが発生する** — サイトにアクセスすると、関係のない外部サイト（アダルトサイト、フィッシングサイト、医薬品の違法販売サイト等）に転送される。特定のデバイス（スマートフォンのみ）や特定の条件（Googleからの流入のみ）でのみリダイレクトが起きるケースもあります。
4. **管理画面にログインできない** — 正しいIDとパスワードを入力してもログインが拒否される場合、攻撃者がパスワードを変更した可能性があります。
5. **知らない管理者アカウントが存在する** — 「ユーザー一覧」に見覚えのないアカウント、特に管理者権限を持つアカウントが追加されている。
6. **FTPやファイルマネージャで不審なファイルを確認できる** — WordPressのコアファイルと無関係な `.php` ファイルがサーバーに置かれている。既存のファイルの更新日時が最終更新作業日と一致しない。
7. **サイトの表示速度が急激に低下した** — 攻撃者がサーバーを踏み台にして別の攻撃を実行している場合、サーバーリソースが消費され表示速度が落ちることがあります。

### H3-2-2: Googleサーチコンソール・GSBからの警告

サイト管理者自身が気づく前に、Googleからの通知で発覚するケースも多くあります。

**Googleサーチコンソール（Google Search Console）**
Googleは、検索クローラーがクロール中にマルウェアや不正コンテンツを検出した場合、サーチコンソールのメッセージセンターに「セキュリティの問題」として通知を送ります。サーチコンソールに登録していれば、異常をいち早くメールで受け取ることができます。

**Googleセーフブラウジング（Google Safe Browsing / GSB）**
Googleのセーフブラウジングはフィッシングサイトやマルウェア配布サイトをデータベースで管理しており、該当サイトへのアクセス時にChromeなどのブラウザ上で「このサイトは危険なサイトとして報告されています」といった赤い警告画面を表示します。この警告が出ると、サイトへの新規訪問者はほぼゼロになり、ブランドへのダメージも甚大です。

Googleセーフブラウジングの診断ツール（`https://transparencyreport.google.com/safe-browsing/search` でURLを入力して確認可能）で、自サイトのステータスを定期的に確認することを推奨します。

### H3-2-3: ホスティング会社・セキュリティスキャナからの通知

**ホスティング会社からのアカウント停止・警告メール**
サーバー上でマルウェアが検出された場合や、自サイトが他サイトへのスパム送信や攻撃の踏み台になっている場合、ホスティング会社が自動検知してアカウントを一時停止し、警告メールを送信するケースがあります。突然サイトが表示されなくなったときは、ホスティング会社のダッシュボードやメールを必ず確認してください。

**Wordfence・Sucuri等のセキュリティプラグイン**
WordPressにインストールするセキュリティプラグイン（Wordfence Security、Sucuri Security等）は、ファイルの整合性チェックやマルウェアスキャンを定期的に実行し、異常を検知した場合にメール通知を送ります。これらのプラグインを未導入の場合、改ざんに気づくタイミングが大幅に遅れる可能性があります。

また、外部のWebサイトスキャンサービス（例：Sucuri SiteCheck）を使ってURLを入力するだけで、マルウェアの有無やブラックリスト登録状況を無料でチェックすることも可能です。定期的な外部スキャンを日常業務に組み込むことが、早期発見の鍵となります。

## H2-3: 被害パターン1｜サイト改ざん（コンテンツ書き換え・スパムリンク挿入）

サイト改ざんは、WordPressを狙った攻撃の中でもっとも「目に見える」被害です。攻撃者がサイトの本文・画像・リンクを書き換えることで、訪問者は知らぬ間に詐欺サイトへ誘導され、運営者は信頼を一夜にして失うことになります。

### H3-3-1: どのような改ざんが行われるか

サイト改ざんは大きく3つの手口に分類できます。

**（1）コンテンツの直接書き換え**
攻撃者がWordPressの管理者権限を奪取した後、または直接サーバー上のファイルを編集することで発生します。トップページのビジュアルや、記事本文のテキストが政治的メッセージや別の言語のプロパガンダに差し替えられる「デフェイス」がこれにあたります。

**（2）スパムリンクの不正挿入**
被害として最も多いのがこのパターンです。攻撃者は、WordPressのデータベース（`wp_posts`テーブルや`wp_options`テーブル）に対して直接SQL文を流し込み、既存の記事本文の末尾や、フッターテンプレートの中に、検索エンジン向けのスパムリンクを大量挿入します。CSSで`display:none`にして訪問者からは見えないようにしつつ、検索エンジンのクローラーにだけ読ませる「クローキング」という手法が多用されます。

**（3）悪意あるJavaScriptの埋め込み（Webスキミング）**
WooCommerceなどのECサイトで特に深刻な改ざん手法です。チェックアウトページに悪意あるJavaScriptを1行挿入するだけで、顧客が入力したクレジットカード番号・有効期限・CVVが攻撃者のサーバーへ送信されます。Magecart攻撃とも呼ばれ、被害が発覚するまでの間、何十件・何百件もの決済情報が漏洩し続けることになります。

**（4）バックドアの埋め込み**
改ざん後に攻撃者が再侵入できるよう、PHPファイルの中にWebシェル（リモートコマンドを実行できる隠しスクリプト）を仕込みます。一見きれいに修復したように見えても、バックドアが残っていれば数日以内に再改ざんされるケースが後を絶ちません。

### H3-3-3: 運営者が得るべき教訓

- **プラグインの更新を72時間以内に適用する**: 上記のFusion Builder・Piotnet Addonsのような CVSS 10.0 の脆弱性は、公開直後から自動スキャナーによる探索が始まります。「後で更新しよう」という判断が侵害の直接原因になります。
- **プラグインの供給元（更新インフラ）まで疑う**: EssentialPluginとSmart Slider 3 Proの事例が示すように、プラグインを最新版に保っていても、開発者側のサーバーが侵害されれば「正規の更新」としてバックドアが配布されます。更新直後は必ずファイルの差分をチェックするか、WordFenceなどのファイル整合性監視ツールを使用してください。
- **ECサイトはチェックアウトページのJavaScript変更を常時監視する**: Webスキミングはログにほとんど痕跡が残りません。Content Security Policy（CSP）を設定し、外部ドメインへのJavaScript通信をホワイトリスト制御することが有効です。
- **バックドア除去には専門ツールを使う**: 視認できる改ざんを修復しても、Webシェルが残れば再侵害されます。修復後は必ずWP-CLI + Wordfence / Sucuri の整合性チェックを実行し、すべての管理者パスワードとAPIキーを再発行してください。

## H2-4: 被害パターン2｜個人情報・顧客情報の漏洩

サイト改ざんは「見た目の被害」ですが、個人情報漏洩は「見えない被害」です。攻撃が成功してからデータが窃取されるまでの間、運営者はまったく気づかないまま顧客情報が流出し続けるケースが珍しくありません。発覚したときには既に数万件の情報が外部に渡っている、というのがこのパターンの最大の恐ろしさです。

### H3-4-1: どのような情報が流出するか

WordPressサイトから漏洩する個人情報は、サイトの性質によって大きく異なります。

| サイトの種別 | 流出しうる情報の例 |
|---|---|
| 一般コーポレートサイト・ブログ | コメント投稿者のメールアドレス、IPアドレス、ユーザー登録情報 |
| EC・WooCommerceサイト | 氏名、住所、電話番号、メールアドレス、注文履歴、クレジットカード情報（スキミングによる場合を含む） |
| 会員制サイト・学習サービス | ユーザーID、パスワードハッシュ、プロフィール情報、購買・受講履歴 |
| お問い合わせフォームを持つサイト | 相談内容、氏名、メールアドレス、電話番号 |
| 医療・福祉関係サイト | 健康情報、診断内容、予約情報（特定の要配慮個人情報） |

特に深刻なのは、**WordPressのデータベース丸ごとの流出**です。SQLインジェクション（SQLi: データベースへの不正なSQL文の挿入）やバックアップファイルの不正取得により、`wp_users`テーブル（全ユーザーのメールアドレスとパスワードハッシュ）や`wp_usermeta`テーブルが一括で盗まれます。

### H3-4-3: 運営者が得るべき教訓

- **フォーム・クイズ・アンケート系プラグインのSQLiに注意する**: QSM（CVE-2025-67987）の事例のように、ユーザーからの入力を受け取る機能を持つプラグインは、SQLインジェクションの標的になりやすい。使用しているプラグインが受け取るパラメータをエスケープ・バリデーションしているか、リリースノートで確認する習慣を持ってください。
- **バックアップファイルを公開ディレクトリに置かない**: WordPressのデータベースバックアップ（`.sql`ファイル）や設定ファイルのバックアップ（`wp-config.php.bak`など）が公開ディレクトリに残っていると、URLを推測するだけで全情報を持ち去られます。バックアップは必ずサーバー外部のストレージに保管し、`.htaccess`で直接アクセスを禁止してください。
- **漏洩を前提とした「通知体制」を事前に整える**: 個人情報保護法の改正（2022年施行）により、個人データの漏洩が発生した場合は個人情報保護委員会への報告と本人通知が義務化されています。発覚後に体制を作るのでは間に合いません。インシデント対応フロー（誰が判断し・誰に通知し・何を公表するか）を平時から文書化しておくことが求められます。
- **ECサイトはPCI DSSの自己問診（SAQ）を定期的に実施する**: WooCommerceで決済を扱う場合、クレジットカードのスキミング攻撃に対する防御として、決済ページのContent Security Policyの設定と、第三者によるJavaScript変更検知の仕組みを導入することを検討してください。

## H2-5: 被害パターン3｜管理者認証情報の窃取・不正ログイン

WordPressの管理画面（`/wp-admin`）に不正ログインされることは、サイトへの「鍵を渡す」ことと同義です。管理者権限さえ手に入れば、攻撃者はプラグインの停止・削除、ファイルの書き換え、バックドアの設置など、あらゆる操作を「正規ユーザーとして」実行できます。IPA「情報セキュリティ10大脅威2024」でも「インターネット上のサービスへの不正ログイン」が9年連続でランクインしており、この脅威がいかに継続的かを示しています。

### H3-5-1: どのように認証情報が窃取されるか

認証情報の窃取には、大きく4つの経路があります。

**（1）ブルートフォース攻撃（総当たり攻撃）**
`/wp-login.php`に対して、IDとパスワードの組み合わせを機械的に試し続ける攻撃です。デフォルトでWordPressはログイン試行回数に制限がないため、対策なしでは「password123」「admin」「12345」のような弱いパスワードは数分で突破されます。

**（2）クレデンシャルスタッフィング**
クレデンシャルスタッフィングとは、過去に別のサービスで流出したID・パスワードの組み合わせリストを、そのままWordPressログインに試す攻撃です。「複数のサービスで同じパスワードを使い回している」ユーザーが標的になります。Have I Been Pwned などのデータによれば、流出済みパスワードのリストは数十億件に達しており、このリストを使った自動化ツールは広く出回っています。

**（3）認証バイパス脆弱性の悪用**
プラグインやテーマのコードの欠陥を利用して、パスワードなしで管理者としてログインする攻撃です。これが最も危険で、弱いパスワードを設定していなくても被害に遭います。

**（4）フィッシング・ソーシャルエンジニアリング**
WordPress管理者を装ったメール（「あなたのサイトに問題が発生しました。こちらからログインして確認してください」など）に偽ログインフォームのリンクを貼り、管理者のパスワードを直接入力させる手口です。

### H3-5-3: 運営者が得るべき教訓

- **二要素認証（2FA）を管理者・編集者アカウント全員に必須化する**: ブルートフォース・クレデンシャルスタッフィングに対して、二要素認証（TOTP / SMSコード）は最も費用対効果の高い対策です。WP 2FA や Google Authenticator などのプラグインを使えば数分で導入できます。パスワードが漏洩していても、2FAがあれば不正ログインを防げます。
- **`/wp-login.php`へのアクセスを制限する**: IPアドレスの許可リスト設定（`.htaccess`またはnginx設定で日本国内からのアクセスのみ許可するなど）、またはログイン試行回数の上限設定（Limit Login Attempts Reloadedなど）を導入してください。Burst StatisticsやAccount Switcherのような認証バイパスは、そもそもログインエンドポイントへのアクセスを絞り込むことで攻撃対象面を減らせます。
- **管理者アカウントのユーザー名を「admin」にしない**: WordPressのデフォルトユーザー名「admin」は、ブルートフォース攻撃の辞書に必ず含まれています。インストール時に別のユーザー名を設定し、既存の「admin」ユーザーは削除または権限を降格してください。
- **プラグインの認証バイパス脆弱性は「パスワードが強くても防げない」ことを認識する**: Account SwitcherやBurst Statisticsの事例が示すように、攻撃者はパスワードを知らなくてもコードの欠陥を突いて管理者になれます。定期的な脆弱性スキャン（WPScan CLI / Wordfence Scan）と、脆弱性情報のサブスクリプション（Wordfence メーリングリスト / Patchstack Alerts）を活用し、使用中のプラグインに関する警告を見逃さない体制を整えてください。

## H2-6: 被害パターン4｜マルウェア・ウイルスの埋め込み

WordPressへの不正アクセスが成功した後、攻撃者が次に行うのが「マルウェアの埋め込み」です。この段階になると被害はサイト運営者だけにとどまらず、そのサイトを訪れた一般ユーザーにまで拡大します。サイトが「加害者の踏み台」になる構造を正確に理解することが、対策の第一歩です。

### H3-6-1: 訪問者にまで被害が広がるマルウェア感染の仕組み

#### マルウェア埋め込みとはなにか

「マルウェア埋め込み」とは、攻撃者が侵入後にWordPressのPHPファイル・JavaScriptファイル・データベースに悪意あるコードを挿入し、そのサイトを訪問したユーザーをさらに攻撃するために利用する手口です。サイト自体は外見上は正常に見えることが多く、運営者自身が気づかないまま数週間から数ヶ月にわたって感染状態が続くケースも少なくありません。

#### 代表的な感染経路と攻撃の連鎖

マルウェアがWordPressサイトに到達する主な経路は次の3つです。

| 感染経路 | 具体的な侵入手段 |
|---|---|
| プラグイン・テーマの脆弱性 | 未パッチのSQLインジェクション / XSS / ファイルアップロード欠陥を悪用してPHPシェルを設置 |
| 認証情報の窃取 | ブルートフォース攻撃・フィッシング・別サービスからの流出パスワードを使った管理画面への不正ログイン |
| サプライチェーン侵害 | プラグイン配布インフラへの不正アクセスにより、公式アップデートとしてマルウェアを配布 |

#### 訪問者への被害が広がる3つの仕組み

**1. ドライブバイダウンロード（Drive-by Download）**
攻撃者が埋め込んだJavaScriptが、訪問者のブラウザ上で自動実行されます。脆弱なブラウザプラグイン（PDF閲覧ソフト・Flash等）を悪用してマルウェアをダウンロードさせる仕組みです。ユーザーはリンクをクリックするどころか、ページを開くだけで感染する可能性があります。

**2. クレジットカードスキマー（Magecart型攻撃）**
WooCommerceなどのECチェックアウト画面に数行のJavaScriptを注入し、フォームに入力されたクレジットカード番号・セキュリティコード・氏名を攻撃者のサーバーにリアルタイム送信します。入力データ自体を横取りするため、TLS暗号化が施されていても防げません。

**3. ClearFake / ClickFix型ソーシャルエンジニアリング**
侵害されたWordPressサイト上に「ブラウザを更新してください」「このエラーを修正するには以下の手順を実行してください」といった偽の警告バナーを表示させ、訪問者に悪意あるスクリプト（PowerShellコマンド等）を自発的に実行させる手法です。Bleeping Computer が2024年に報告した事例では、この手法が6,000以上のWordPressサイトで実行されました（出典: Bleeping Computer, 2024年）。

**4. バックドアによる持続的侵害**
攻撃者はコードを一度埋め込むだけでなく、「隠匿管理者アカウントの作成」「複数箇所へのバックドアファイルの分散設置」「C2（コマンド&コントロール）サーバーとの通信確立」を行い、クリーンアップ後も再感染できる状態を維持します。

### H3-6-3: 運営者が得るべき教訓

上記の事例から、運営者が実務として取り組むべき対策を整理します。

- **プラグインの購入・更新元を必ず確認する。** サプライチェーン攻撃では「公式の更新チャネル」そのものが汚染されます。プラグイン売買プラットフォーム経由で所有者が変わったプラグインは特に注意が必要です。WordPress.org の更新履歴とコミット差分（プラグインのSVNリポジトリ）を確認する習慣が有効です。
- **管理者アカウントを定期的に棚卸しする。** Smart Slider 3 Proの事例では、`wpsvc_*` パターンの隠匿管理者アカウントが一覧から非表示化されていました。`wp_users` テーブルを直接確認するか、WP-CLIで全ユーザーを書き出して不審なアカウントがないかチェックしてください。コマンド例: `wp user list --fields=ID,user_login,user_email,roles`
- **ファイルの整合性を監視する。** WordPressのコアファイル・プラグインファイルが改ざんされた場合に検知できるファイル整合性監視（File Integrity Monitoring）ツールを導入してください。Wordfence・iThemes Security等のプラグインにこの機能が含まれます。
- **ECサイト運営者はチェックアウト画面の外部スクリプトを厳格に管理する。** クレジットカードスキマーは数行のJavaScriptで実現されます。Content Security Policy（CSP）ヘッダでスクリプトの読み込み元を許可リスト制御することで、未知の外部スクリプトの実行を防げます。

## H2-7: 被害パターン5｜DDoS攻撃・サービス妨害（DoS）

サイト改ざんは「コンテンツを書き換える」攻撃ですが、DDoS（Distributed Denial of Service）攻撃は「サービスそのものを停止させる」攻撃です。表示回数ゼロ・売上ゼロが続く状態は、改ざんと同等かそれ以上の事業損害をもたらします。また、WordPressサイト自体が「攻撃の踏み台」として悪用されるケースも多く、知らないうちに加害者側になるリスクも存在します。

### H3-7-1: サイトダウンを引き起こす攻撃の仕組み

#### DDoSとDoSの違い

- **DoS（Denial of Service）**: 単一の送信元から大量リクエストを送りサービスを停止させる攻撃。
- **DDoS（Distributed DoS）**: 世界中に散らばる多数の感染端末（ボットネット）から同時に大量リクエストを送る攻撃。送信元が分散しているためIPブロックでの防御が困難です。

#### WordPressに関連する主な攻撃手法

**1. Layer 7（アプリケーション層）HTTP Flood**
Webサーバーに対して大量のHTTPリクエストを送り、PHPの実行・データベースアクセスを過負荷にしてサイトを停止させます。正規ユーザーのリクエストと見かけ上区別がつきにくい点が特徴です。Cloudflareの2024年Q4レポートによれば、HTTP DDoS攻撃全体の51%はHTTP攻撃で、73%が既知のボットネットによるものです（出典: Cloudflare DDoS Threat Report Q4 2024）。

**2. XML-RPC を悪用したリフレクション・増幅攻撃**
WordPressには `xmlrpc.php` というXML-RPC APIエンドポイントが存在します（現在は標準で無効化されていますが、古いサイトや一部プラグインが有効化しているケースがあります）。攻撃者はこのエンドポイントを介して「ピンバック（Pingback）」機能を悪用し、第三者のWordPressサイトを踏み台にして標的サイトに大量リクエストを集中させます。運営者が気づかぬうちに自サイトが攻撃の「踏み台」になるパターンです。

**3. SYN Flood / UDP Flood（ネットワーク層攻撃）**
TCP接続の確立処理（SYNパケット）やUDPパケットを大量送出し、サーバーのネットワーク帯域・接続テーブルを枯渇させます。Cloudflareが2024年10月に観測した史上最大5.6 Tbpsの攻撃はMirai亜種ボットネットによるUDPフラッドであり、13,000以上のIoTデバイスが動員されました（出典: Cloudflare DDoS Threat Report Q4 2024）。

**4. Memcached・BitTorrentを悪用した増幅攻撃**
Memcached（分散メモリキャッシュサーバー）はUDPに対応している場合にリフレクション・増幅攻撃（Amplification Attack）の踏み台になります。攻撃パケットを数十〜数万倍に増幅できます。Cloudflare 2024年Q4レポートでは、Memcached DDoS攻撃が対前四半期比314%増を記録しています（出典: 同上）。

**5. 身代金DDoS（RDoS）**
「支払いがなければDDoS攻撃を継続する」と脅迫するランサムウェア型の亜種です。Cloudflareの同レポートでは2024年Q4に顧客の12%がこうした身代金要求を受けており、前四半期比78%増という急増傾向にあります（出典: 同上）。

### H3-7-3: 運営者が得るべき教訓

DDoS対策はサーバーやプラグインの話にとどまらず、「インフラ構成」の問題です。以下の実務的な対策を優先順位の高い順に整理します。

- **CDN・DDoS軽減サービスを導入する。** Cloudflare（無料プランあり）・AWS Shield・さくらのセキュリティオプション等のCDNを経由させることで、大規模フラッド攻撃の大半をサーバーの前段で遮断できます。特に個人・中小規模のWordPressサイトでは、CDN導入が最もコストパフォーマンスの高い対策です。
- **XML-RPC（xmlrpc.php）を無効化する。** WordPress 6.1以降はデフォルトで実質的に制限されていますが、古いサイトや一部プラグインが有効化している場合があります。`.htaccess` でのブロック、または専用プラグイン（Disable XML-RPC等）で無効化してください。自サイトが第三者への攻撃踏み台になるリスクを排除できます。
- **ホスティング契約の帯域・リソース上限を事前に把握しておく。** 低価格共有サーバーでは、DDoS攻撃を受けると同一サーバーの他の利用者にも影響が出るため、ホスティング事業者がサービスを強制停止するケースがあります。VPS・マネージドWPホスティング（Kinsta / WP Engine等）への移行、またはサーバー事業者のDDoS対策サービスの有無を確認してください。
- **RDoS（身代金DDoS）の脅迫を受けた場合は支払わない。** 支払っても攻撃が止まる保証はなく、「支払う企業」として再度標的になる可能性が高まります。事業者（ホスティング・CDN）に即時連絡し、攻撃トラフィックの遮断・レート制限の強化を依頼することを最優先にしてください。

## H2-8: 改ざんを発見したときの対処手順

改ざんに気づいた瞬間、多くの担当者は焦りから場当たり的な対応を取ってしまいがちです。しかし初動を誤ると被害が拡大し、後から原因を追えなくなる恐れがあります。まず深呼吸して、以下の5ステップを順番に実行してください。

### H3-8-1: ステップ1｜まずサイトをオフライン（メンテナンスモード）にする

**目的**: 改ざんされたサイトを公開し続けると、訪問者にマルウェアが配布され被害が拡大します。また攻撃者によるさらなる改ざんや情報搾取が継続するリスクもあります。**最初にすべきことはサイトを止めること**です。

**具体的な作業**

方法1: `.htaccess` で一時的にアクセスをブロックする（Apache環境の場合）

```apache
# .htaccess の先頭に追記（自分のIPだけ通す）
Order deny,allow
Deny from all
Allow from XXX.XXX.XXX.XXX  # 自分の固定IPアドレスに置き換える
```

方法2: WP-CLI でメンテナンスモードを有効化する

```bash
wp maintenance-mode activate --allow-root
```

サーバーがCloudflareなどのCDN経由の場合は、CDNコントロールパネルから「オリジンへのトラフィックを停止」または「Under Attack Mode」を有効にすることも有効です。

**使えるツール**: ホスティングのコントロールパネル / Cloudflare の「Under Attack Mode」 / WP-CLI（SSH接続が可能な場合）

**やってはいけないこと**:
- 改ざんに気づいても「少しだけ様子を見る」と公開し続けること（マルウェア配布を継続することになります）
- WordPressの管理画面にログインして何か操作しようとすること（攻撃者にセッション情報を奪われる可能性があります）

### H3-8-2: ステップ2｜バックアップを確保し被害範囲を特定する

**目的**: 改ざんを受けた状態のファイル・データベースを証拠として保全した上で、正常だった時点のバックアップを確認します。「何が変更されたか」を把握しなければ、適切な修復はできません。

**具体的な作業**

まず現状のファイルをバックアップします（後の比較・証拠用）。

```bash
# SSH経由でWordPressルートを丸ごとアーカイブ（例: /var/www/html が対象）
tar -czf /tmp/compromised_backup_$(date +%Y%m%d%H%M).tar.gz /var/www/html

# データベースもダンプ
wp db export /tmp/compromised_db_$(date +%Y%m%d%H%M).sql --allow-root
```

次に、不正変更されたファイルを特定します。

```bash
# 直近24時間以内に変更されたPHPファイルを検索
find /var/www/html -name "*.php" -newer /var/www/html/wp-config.php -mtime -1

# 疑わしい文字列（base64エンコードされたコード埋め込みなど）を検索
grep -r --include="*.php" "base64_decode" /var/www/html
grep -r --include="*.php" "eval(" /var/www/html
```

**使えるツール**: UpdraftPlus などのバックアッププラグイン / WP-CLI の `wp checksum core`

```bash
wp checksum core --allow-root
# 公式チェックサムと一致しないファイルが一覧表示される
```

**やってはいけないこと**:
- 改ざんを受けたままサーバーのファイルを削除・上書きする（証拠が失われます）
- 「正常に見えるから大丈夫」と判断してバックアップ確認を省略する

### H3-8-3: ステップ3｜マルウェア・不正ファイルのスキャンと除去

**目的**: 攻撃者が仕込んだバックドアや悪意あるコードを確実に取り除きます。**表面的に見えるページの修復だけでは不十分**で、ファイルシステム全体をスキャンする必要があります。

**具体的な作業**

```bash
# WordPressコアファイルを再インストール（設定・データベースは変更しない）
wp core download --force --allow-root

# プラグインの整合性チェック（WordPress.org登録プラグインのみ対応）
wp plugin verify-checksums --all --allow-root

# テーマの整合性チェック
wp theme verify-checksums --all --allow-root
```

自作テーマやカスタムプラグインはチェックサム検証ができないため、Git等のバージョン管理と比較して差分を確認します。

**使えるツール**

| ツール | 特徴 |
|---|---|
| Wordfence（プラグイン） | ファイルシステムスキャン、脅威データベースとの照合 |
| Sucuri Security（プラグイン） | マルウェアスキャン＋クラウド型モニタリング |
| MalCare | AIベースのスキャン、ワンクリック除去機能あり |
| WP-CLI `verify-checksums` | 公式コアファイル・プラグインの改ざん検出 |

**やってはいけないこと**:
- スキャンツールが「クリーン」と報告しても、疑わしい`.php`ファイルが残っていたら手動で確認する（スキャナーがすべての脅威を検出できるとは限りません）
- コアファイルを手動で1つずつ上書きする（漏れが出るため、`wp core download --force` を使ってください）

### H3-8-4: ステップ4｜パスワード・認証情報の全リセット

**目的**: 攻撃者はサイトへの侵入経路（認証情報）をすでに入手しています。マルウェアを除去しても認証情報を変更しなければ、**数時間以内に再侵入される**ケースが実際に報告されています。認証に関わるすべての情報をリセットします。

**具体的な作業**

```bash
# 全管理者ユーザーのパスワードを一括更新（WP-CLI）
wp user list --role=administrator --field=ID --allow-root | \
  xargs -I {} wp user update {} --user_pass="$(openssl rand -base64 24)" --allow-root

# 秘密鍵・ソルトの再生成（wp-config.php の AUTH_KEY 等を更新）
wp config shuffle-salts --allow-root
```

その他リセットすべき認証情報のチェックリスト。

- [ ] WordPress 管理者パスワード（全アカウント）
- [ ] FTP / SFTP アカウントのパスワード
- [ ] データベース（MySQL）のパスワード → `wp-config.php` も更新
- [ ] ホスティングコントロールパネルのパスワード
- [ ] 関連するメールアカウントのパスワード
- [ ] APIキー（決済サービス、メール配信サービス等）

**使えるツール**: WP-CLI / パスワードマネージャー（1Password、Bitwarden 等） / ホスティング会社のコントロールパネル

**やってはいけないこと**:
- WordPressの管理者パスワードだけ変えてFTPやDBを変えずに放置する
- 以前と似た文字列（末尾に数字を足しただけなど）の弱いパスワードを設定する

### H3-8-5: ステップ5｜原因の特定・修正後にサイトを公開再開する

**目的**: 「なぜ侵入されたか」を明確にせずにサイトを再公開すると、**同じ手口で再び改ざんされます**。原因を特定・修正してから公開を再開することが不可欠です。

**具体的な作業**

```bash
# Apacheアクセスログから不審なPOSTリクエストを抽出
grep "POST" /var/log/apache2/access.log | grep -v "wp-login.php\|wp-admin" | tail -100

# 不審なファイルアップロードを確認
grep "\.php HTTP" /var/log/apache2/access.log | grep "upload"
```

公開再開前の最終確認事項:
1. 脆弱性のあったプラグイン・テーマが更新または削除されていること
2. スキャンツールがすべてクリーンを報告していること
3. Google Search Console に「セキュリティの問題」レポートが表示されている場合は「審査をリクエスト」を送信済みであること
4. 念のため CDN・WAF のキャッシュをパージしていること

```bash
# WP-CLIでサイトURLとwp-config.phpの整合性を最終確認
wp config get --allow-root
wp site list --allow-root
```

**使えるツール**: Google Search Console / ホスティングの生ログ / Sucuri SiteCheck (`https://sitecheck.sucuri.net/`)

**やってはいけないこと**:
- 「たぶん大丈夫だろう」という感覚でログ確認を省略して再公開する
- 改ざん後のサイトをGoogle Search Consoleで「審査リクエスト」せずに放置する（検索結果に警告が出続けます）

## H2-9: 再発防止のためにいま取り組むべきセキュリティ対策

改ざん対応が完了した後、または「まだ被害を受けたことはないが備えたい」という方にとって、以下の4つの対策は**コストをかけずに実施できる最も効果的な防御ライン**です。攻撃者の多くは高度な標的型ではなく、脆弱性スキャンで無防備なサイトを自動的に探しています。基本対策を徹底するだけで、ターゲットになりにくくなります。

### H3-9-1: プラグイン・テーマ・WordPress本体の定期更新

**なぜ必要か**: WordPressサイトへの攻撃の多くは、既知の脆弱性を持つプラグインやテーマを標的にしています。脆弱性が公表されると、多くの場合その直後に自動スキャン攻撃が始まります。修正パッチを適用するまでの時間が短いほど、リスクは小さくなります。

**具体的な実施方法**

1. **自動更新を有効にする**（マイナーバージョンアップは最低限自動化）

```php
// wp-config.php
define( 'WP_AUTO_UPDATE_CORE', 'minor' ); // マイナーバージョンのみ自動更新
```

プラグインの自動更新は WordPress 管理画面の「プラグイン」一覧から、各プラグインの「自動更新を有効にする」リンクで設定できます。

2. **使っていないプラグイン・テーマは削除する**（無効化ではダメ）

```bash
# 無効化されているプラグインを確認
wp plugin list --status=inactive --allow-root

# 不要プラグインを削除
wp plugin delete プラグイン名 --allow-root
```

3. **月1回、更新状況を点検する定例作業をカレンダーに入れる**

### H3-9-2: 強力な認証設定（二要素認証・ログイン試行制限）

**なぜ必要か**: ブルートフォース攻撃（パスワードの総当たり）やパスワードリスト攻撃（流出済み認証情報の転用）は、WordPressのデフォルト設定のままではほぼ無防備です。認証を強化することで、これらの攻撃の大部分を防げます。

二要素認証（2FA）の導入

| プラグイン | 特徴 |
|---|---|
| WP 2FA | セットアップウィザードが親切、無料版あり |
| Google Authenticator (miniOrange) | Google Authenticator対応、柔軟な設定 |
| Wordfence Login Security | Wordfenceと統合、TOTP対応 |

ログイン試行制限の設定

```apache
# .htaccess でwp-login.phpへのアクセスをIP制限する例
<Files wp-login.php>
  Order deny,allow
  Deny from all
  Allow from XXX.XXX.XXX.XXX  # 自分のIPのみ許可
</Files>
```

その他の認証強化策:
- 管理者ユーザー名を `admin` から別の名前に変更する
- `/wp-admin` へのアクセスを固定IPに制限する
- XML-RPC を使用しない場合は無効化する

### H3-9-3: 定期バックアップと復元テストの実施

**なぜ必要か**: バックアップがあれば、改ざん被害を受けても「クリーンな状態への巻き戻し」が現実的な選択肢になります。しかし**バックアップがあっても復元できなければ意味がありません**。「バックアップを取っているつもり」で実際には壊れていたケースも少なくないため、定期的な復元テストがセットで必要です。

バックアップの推奨設定

| 項目 | 推奨設定 |
|---|---|
| バックアップ頻度 | 毎日（最低週1回） |
| 保存世代数 | 14日分以上 |
| 保存場所 | サーバー外（クラウドストレージなど） |
| 対象 | ファイル + データベースの両方 |

**推奨プラグイン例**

- **UpdraftPlus**: Google Drive / Dropbox / S3 等へのクラウド保存に対応
- **BackWPup**: 多様な保存先に対応、スケジュール設定が柔軟
- **Jetpack Backup**（旧VaultPress）: リアルタイムバックアップ対応（有料）

```bash
# WP-CLIでバックアップを取得する例
wp db export /backup/db_$(date +%Y%m%d).sql --allow-root
tar -czf /backup/files_$(date +%Y%m%d).tar.gz /var/www/html
```

ステージング環境にバックアップを復元し、正常に動作することを**3ヶ月に1回**確認することを推奨します。

### H3-9-4: WAF・セキュリティプラグインの導入

**なぜ必要か**: WAF（Web Application Firewall）は、不正なHTTPリクエストをWordPressが処理する前にブロックします。SQLインジェクションやXSS（クロスサイトスクリプティング）といった代表的な攻撃パターンに対して、プラグインやWordPressの更新だけでは防げない攻撃も遮断できます。

WordPressプラグイン型WAF・セキュリティプラグインの比較

| プラグイン | 特徴 | 費用 |
|---|---|---|
| Wordfence Security | ファイアウォール＋マルウェアスキャン、ログイン保護、無料版で主要機能が使える | 無料 / 有料 |
| Sucuri Security | 無料版はモニタリング中心。有料版はクラウドWAF（CDN経由） | 無料 / 有料 |
| SiteGuard WP Plugin | 日本の開発元（JP-Secure）、日本語環境に最適化、管理画面URLの変更機能など | 無料 |
| iThemes Security | ユーザーフレンドリーな設定画面、ファイル変更検知、2FA対応 | 無料 / 有料 |

**Cloudflare** の無料プランでも基本的なDDoS対策とWAFのルールが利用できます。DNSをCloudflare経由にするだけで導入でき、設定の敷居が低いため中小規模サイトにも現実的な選択肢です。

IPAの「中小企業の情報セキュリティ対策ガイドライン」でも、ウェブサービスの不正アクセス対策として多層防御の考え方が推奨されています（参照: IPA `https://www.ipa.go.jp/security/sme/` ※最新版をご確認ください）。

## H2-10: まとめ｜被害を最小化するための「事前対策」と「事後対応」

WordPressサイトの改ざんは、大企業だけの問題ではありません。むしろ、セキュリティ対策に人的リソースを割けない中小企業や個人運営のサイトほど、自動スキャンによる攻撃のターゲットになりやすいのが現実です。本記事で紹介した内容を振り返ります。

**本記事で解説した5つの被害パターン**

- **サイト改ざん（コンテンツ書き換え・スパムリンク挿入）**: 攻撃者がサイトの表示内容を書き換え、スパムリンクや不審なコンテンツを埋め込む。管理者が気づかない間にSEO評価が毀損され、訪問者の信頼を失う。
- **個人情報・顧客情報の漏洩**: 顧客の氏名・メールアドレス・決済情報などが外部に流出し、個人情報保護法違反や損害賠償請求など法的責任が生じるリスクがある。
- **管理者認証情報の窃取・不正ログイン**: ブルートフォース攻撃やフィッシングによって管理者アカウントが乗っ取られ、攻撃者がバックドアを設置して永続的な侵入経路を確保する。
- **マルウェア・ウイルスの埋め込み**: サイトを踏み台にして訪問者の端末に不正スクリプトを感染させ、Googleのセーフブラウジング警告によって事実上アクセス不能な状態に追い込まれる。
- **DDoS攻撃・サービス妨害（DoS）**: 大量の不正リクエストでサーバーを過負荷状態にし、サイトをダウンさせる。機会損失だけでなく、ブランド信頼性への打撃も深刻になる。

**「事前対策」と「事後対応」の対比**

| 観点 | 事前対策 | 事後対応 |
|---|---|---|
| 目的 | 攻撃を受けにくくする／被害を最小化する | 被害を止め、正常な状態に復旧する |
| コスト | 低い（主に時間・設定作業） | 高い（対応工数・信用回復・機会損失） |
| 主な施策 | 定期更新・強力な認証・WAF・バックアップ | オフライン化→スキャン→除去→リセット→再開 |
| 効果の見えやすさ | 「何も起きない」ため実感しにくい | 被害が止まることで効果が明確 |
| 取り組むべきタイミング | 今すぐ | 改ざん発覚直後、1分でも早く |

事後対応は「起きてから対応する」ため、どうしても費用・工数・信用面での損失が伴います。一方で事前対策は、プラグインの自動更新設定やセキュリティプラグインの導入など、**数時間の作業で長期的なリスクを大幅に下げられる**ものがほとんどです。

**今すぐできる次のアクション**

本記事をお読みの方の中には「自分のサイトが今どういう状態か、正直わからない」という方も多いのではないでしょうか。まずは現状を把握することが第一歩です。

- **外部スキャンによる無料診断**: Sucuri の SiteCheck（`https://sitecheck.sucuri.net/`）にURLを入力するだけで、外部からのマルウェア感染やブラックリスト掲載状況を確認できます
- **Google Search Console の「セキュリティの問題」を確認**: 登録済みのサイトであれば、Googleが検出した問題を無料で確認できます
- **より詳しい診断・対応支援**: 改ざんの疑いがある、または社内リソースだけでは対応が難しいと感じる場合は、専門家への相談が最も確実な解決策です。wpsecurity.jp では、WordPressサイトのセキュリティ診断および改ざん対応支援をご提供しています。お気軽にお問い合わせください
